# Slide 10: Context isolation

**Part 1: SUBAGENTS 개념과 원리**

## Code Blocks

### Context Model

```
Main Agent Context (200K tokens)
┌─────────────────────────────────────────────┐
│ User: 이 코드베이스의 보안 취약점을 점검해 줘 │
│ Main: 분석을 시작하겠습니다.                 │
│ Main: → Task(security-scanner, ...)         │
│                                              │
│  ┌──────────────────────────────────────┐   │
│  │ Subagent Context (별도 200K)        │   │
│  │ - 시스템 프롬프트 (security 전문)    │   │
│  │ - 파일 100개 분석 (수십만 토큰)      │   │
│  │ - 분석 종료 후 요약만 반환           │   │
│  └──────────────────────────────────────┘   │
│                                              │
│ Main: 결과 요약: 5건의 high severity 발견.   │
│ User: 첫 번째 항목을 자세히 설명해 줘        │
└─────────────────────────────────────────────┘
```

## Speaker Notes

Context Isolation을 다이어그램으로 살펴봅니다.
메인 에이전트는 자신의 200K 토큰 윈도우를 가집니다.
사용자와의 대화와 작업의 큰 그림이 여기에 누적됩니다.
Subagent를 호출하면 별도의 200K 토큰 윈도우가 새로 생성됩니다.
서브의 시스템 프롬프트, 분석 대상 파일들, 중간 추론 과정이 모두 이 별도 컨텍스트에 들어갑니다.
예시처럼 파일 100개 분석에 수십만 토큰을 쓰더라도 메인은 영향을 받지 않습니다.
서브가 끝나면 요약 결과만 메인에 반환됩니다.
메인은 요약만 받아 사용자에게 전달하고 다음 대화를 이어갈 수 있습니다.
이 구조 덕분에 큰 작업을 처리해도 메인의 대화 흐름이 보존됩니다.
